Vault load balancer downtime while Vault nodes stay healthy

Why hash-vault.koneksi.co.kr can fail even when all Vault cluster members are up — and how that differs from observing IPFS LB stability.

How Vault is exposed (no nginx on Vault nodes)

Vault servers do not run nginx. Each node listens on a public IP and port (typically :8200) — clients can talk to Vault directly on https://<node-ip>:8200 (or HTTP, depending on your listener config).

The file hash-vault-lb.conf is optional: it runs on a different host (e.g. node-5) and only affects traffic that uses the DNS name hash-vault.koneksi.co.kr. That machine is nginx-only in front of the three Vault IPs — it is not part of the Vault process on the cluster members.

Two paths. (1) Direct: IP:8200 → Vault only. (2) Hostname: browser/CLI to hash-vault.koneksi.co.kr → nginx LB → Vault backends. LB outages hit path (2) only; path (1) can still work if firewalls allow it.

Core idea

Vault runs as a multi-node cluster (e.g. three servers, each with Vault on port 8200). Any central load balancer (nginx on node-5) is a separate layer that terminates TLS and proxies to those node IPs — it does not run on the Vault VMs themselves.

Nodes up ≠ LB path up. Cluster health only proves Vault processes and raft are fine. Clients using the public hostname hit nginx first; if that layer fails, users see downtime even though every Vault node passes vault status.

Why the Vault LB can fail alone

  • Single point of failure: One nginx host, one TLS cert lifecycle, one set of firewall rules for the VIP/DNS target.
  • Process / host issues: nginx reload error, OOM, disk full, kernel patch reboot — Vault nodes unaffected.
  • Passive upstream health: Standard nginx marks peers bad only after failed requests (max_fails / fail_timeout), not proactive health probes.
  • Vault-specific responses: Sealed member, standby redirect, leader election, or long operations can produce 502/504 or client errors that feel like “LB down” while the cluster still considers nodes “up”.
  • Timeouts and keepalives: proxy_read_timeout and HTTP keepalive to backends can cause brief errors under load or after idle connection reuse.

Vault LB vs IPFS LB (why IPFS may look “always up”)

Topic Vault LB IPFS LB (typical)
Traffic shape API/UI, auth, sometimes long requests; sensitive to leader and seal state. Often more cacheable or retried; different error visibility.
Observation Every UI/API failure is obvious to operators. May be measured differently or used by fewer interactive clients.
Hostname / cert hash-vault.koneksi.co.kr has its own cert and vhost lifecycle. Different vhost; incidents do not always coincide.

Improvements (if you need higher availability)

  • Second LB or VIP: e.g. keepalived, cloud load balancer, or DNS failover between two nginx instances.
  • Active health checks: Nginx Plus, or an external checker hitting /v1/sys/health and updating upstream state.
  • Dedicated LB VM for Vault so it does not compete with heavy OpenResty/IPFS on the same box.
  • Logs: Correlate incidents with error.log (connect() failed, upstream timed out, no live upstreams).

C) What happens to traffic when the LB layer fails

Vault cluster healthy, nginx LB host unhealthy: traffic that only uses the DNS name hits a dead or flaky edge — timeouts, TLS errors, or 502. Traffic that uses node public IP and :8200 (and is allowed by firewall) bypasses nginx and still reaches Vault.
